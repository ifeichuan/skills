# 编码前思考框架

代码不是工程的起点，而是工程的结果。写代码之前停几分钟，把不确定的想明白再动手——这不是磨蹭，是判断力。

## 核心原则

动手前问自己这 10 个问题：

1. **用户真正要解决的是啥？** —— 需求单上写的不一定是真问题，把问题问具体
2. **有没有现成写法？** —— 先翻项目里类似功能别人怎么写的，保持一致
3. **改动到哪儿为止？** —— 画死边界，不该碰的坚决不碰
4. **真实数据长啥样？** —— 别按理想情况写代码，线上数据什么妖魔鬼怪都有
5. **最致命的出错是哪个？** —— 先想"怎么崩"，再想"正常怎么走"
6. **改了会牵连到谁？** —— 你的改动可能被安卓、报表、客服系统偷偷依赖
7. **靠啥证明做成了？** —— 「本地点了下是好的」不是证据，要可验证的标准
8. **最小靠谱版本长啥样？** —— 别一上来盖大楼，先盖不会塌能住人的小房子
9. **PR 大概啥样？** —— 脑子里预演一遍，包括决定不做什么
10. **搞砸了撤得回来吗？** —— 不是所有错都能回滚，提前想退路

## 模式选择

根据用户的意图选择对应模式，读取对应的 reference 文件：

| 用户意图 | 模式 | Reference |
|---------|------|-----------|
| 准备动手写代码，想逐条过一遍 | **checklist** | `references/checklist.md` |
| 给了一个需求/PRD，要我帮忙分析 | **analyze** | `references/analyze.md` |
| 描述了要做的改动，想预估范围 | **preview** | `references/preview.md` |
| 思路不清晰，需要引导理清 | **mentor** | `references/mentor.md` |

如果用户没有明确指定模式，根据语气和上下文判断：
- 用户带着具体需求描述来的 → `analyze`
- 用户说「准备开始」「要动手了」→ `checklist`
- 用户描述的是代码改动本身 → `preview`
- 用户说「有点乱」「不知道从哪下手」→ `mentor`

选定模式后，**先读取对应 reference 文件**，再按其中的指引执行。
